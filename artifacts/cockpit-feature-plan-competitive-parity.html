02

Every blocker, resolved to a verdict

Five capabilities are structurally out of Cockpit's reach the incumbent way — no OTA API partnership, no channel-manager API. Each one got researched for a real pivot, not a hopeful one. Three sub-findings came back honestly NOT-VIABLE rather than forcing an answer.

1. Two-way channel sync (pushing rates/availability out)

Not viable to replicate

Chosen pivotRead-only sidecar next to the host's existing publisher — already the architecture. Real precedent: AirDNA's public-data model; Turno's own iCal-vs-API-partner tiering.

What it costs usiCal gives availability blocking only — one-way, asymmetric, and throttled (Airbnb auto-refreshes every ~3hrs with an undocumented request cap; Vrbo caps at 30min / 5 calendars). A price-push via a human-triggered co-host session has no real product doing it anywhere, and collides directly with Airbnb's ToS §11.1 bot ban, which carries no co-host carve-out.

Honest framing for hostsCockpit shows the truth; it never becomes a second system that can cause a double-booking — which every incumbent's own users complain about.

2. Payments (taking a cut, processing bookings)

Viable pivot

Chosen pivotFlat SaaS, no payment rails, plus Plaid bank-feed read for reconciliation — proven twice independently (Baselane, Clearing), both explicit that funds never leave the originating institution via the dashboard. Stripe Connect stays scoped to a future Cockpit-owned direct-booking channel only, never as an OTA-payments substitute.

3. Real-time data without webhooks

Viable, with an honest ceiling

Chosen pivotGmail push (watch + Pub/Sub) delivers a new email event in 1–10 seconds once it lands — the email transport itself is not the bottleneck. The real floorUnmeasured OTA send-latency for the email itself, and iCal's documented 30-minute-to-3-hour ceiling for anything that only shows up via calendar sync. No non-partner tool beats this. Set honest latency expectations rather than promising webhook parity.

4. Writing back — replying to guests, changing prices

Draft-only viable; price write-back not viable

Chosen pivotDraft-and-insert via a browser extension, human clicks the platform's own send button. Real and shipping today (HostReply AI, AI Host Assistant for Airbnb — both confirmed draft-only, both explicitly non-affiliated with Airbnb). This sits in a genuine ToS gray zone under the same §11.1 clause with no confirmed enforcement either direction — an instructional, not structural, guarantee, which means a silent-auto-send feature creeping in later would be invisible until it isn't. Draft-only, forever, is the line. Price-field write-back has no real precedent anywhere — not viable, don't build it.

5. Screening, smart locks, reviews

Locks & screening viable · reviews not viable

Locks & screeningFully independent of OTA-API-partner status — Seam and RemoteLock for locks, Autohost for a standalone guest-verification link that works "without integrating Autohost at all." Freely buildable now.

ReviewsThe honest failure case. No confirmed public Airbnb Reviews API for non-partners, and even Revyoos (a review-aggregation product) discloses no sourcing mechanism when its own page is opened directly. Not building review automation rather than inventing an unconfirmed pivot.

03

What we're deliberately not building

Naming these explicitly so nobody promises them later by accident — each one was researched, not assumed away.

  • ✕Automated price-push to any OTA via any mechanism — no precedent exists, direct ToS collision.
  • ✕Silent auto-send of guest replies — draft-and-human-click only, permanently.
  • ✕Native review sourcing/automation — no confirmed non-partner access to Airbnb's review data.
  • ✕Becoming a payment processor — Plaid read-only reconciliation instead.
  • ✕Two-way channel distribution — read-only sidecar to the host's existing publisher, by design.
04

The wedge — a feature spec nobody else has

This isn't "PadSplit is underserved" as a slogan — it's a concrete build spec. A PadSplit host leases rooms, not properties; members pay weekly/biweekly while payout lands monthly, in arrears; PadSplit's own dashboard confirmed to give only property- and portfolio-level views, never per-room. No STR PMS surveyed models a room as the leasable unit at all, and none of the four coliving-adjacent tools found (ColivHQ, COHO, Bidrento, MonkSpaces) serve PadSplit specifically. That gap is buildable, and it's the real differentiated core of Cockpit.

01

Room-level financial roll-up

No STR PMS models the room as the leasable unit — PadSplit's own dashboard confirms only property/portfolio aggregates.

02

Independent arrears / AR-aging ledger

A real host distrusts PadSplit's own 97%-collection claim on BiggerPockets — an audited ledger cross-checked against the platform's self-reported numbers is a directly evidenced need.

03

Payout reconciliation

Bridges the weekly/biweekly member-billing vs. monthly-host-payout mismatch. PadSplit gives a raw CSV; it doesn't reconcile it.

04

Per-room P&L

Revenue minus allocated utility, turnover, and vacancy-day cost — confirmed absent everywhere surveyed, PadSplit included.

05

Unified member-lifecycle funnel

PadSplit has the discrete pieces (applied → approved → moved-in → late → terminated); ColivHQ sells exactly this pipeline in the adjacent category, evidencing real demand.

06

Turnover-cost tracking

Tied to churn/rating data PadSplit already collects but never surfaces financially.

07

Segment-aware ingestion

A structural fork, not a detail: PadSplit-CSV/email-centric vs. bank/Baselane-centric for MTR — Furnished Finder holds no transaction record at all.

08

Portfolio-wide reputation visibility

PadSplit's Member Score is explicitly cross-host and portable by design — no confirmed portfolio roll-up view exists anywhere to track it.

09

Utility cost-per-room allocation

PadSplit puts overage risk on the host with no tool; Bidrento's cost-allocation feature is the direct precedent for what's missing.

10

MTR deposit ledger + vacancy forecasting

Furnished Finder hosts hold their own deposits with state-specific return deadlines, and vacancy risk clusters around known ~13-week assignment-end dates — no tool found models either.

Data-readiness — split honestly, not all ten are equal

7 of 10 (including #1, #2, #3, #5, #7, #8, #10) are reachable purely on the already-chosen data path — PadSplit's host-facing collections/termination emails (verified), the transaction-level CSV export, and Furnished Finder hosts' own bank/Baselane feed via Plaid. Three are not yet data-ready and need more than access: #4 (per-room P&L) and #9 (utility cost-per-room allocation) both require allocating a utility bill down to the room — that dollar figure isn't sourced anywhere in email, CSV, or Plaid today, so this needs either manual entry or a new utility-provider integration, not just an ingestion pipe. #6 (turnover-cost tracking) is worse: PadSplit's own "Needs Flip" status is a label, not a cost-tracked event — there's no cost data behind it anywhere found. Build #4/#6/#9 as Phase 2, once a cost-capture mechanism exists, not alongside the seven that are ready now.

05

The specialist layer — how we earn credibility as a cockpit, not a toy PMS

Cockpit sits alongside a host's existing setup, the way KeyData does — so the features that matter most are the ones specialist analytics/ops tools nail and full PMSs treat as an afterthought.

  • →The verified-data trust argument — KeyData's own positioning: platform-sourced data, not scraped or modeled, is the credibility bar for any read-only analytics layer.
  • →Pacing & forecasting — booking-pace trend lines against prior periods; a table-stakes requirement for a credible analytics layer, not an optional nicety.
  • →Trust accounting — only 2 of 8 incumbents name it explicitly; a real differentiator, not table stakes, and squarely inside the "financial truth" wedge already chosen.
  • →An opinionated derived health metric — one number per property/room that rolls up occupancy, arrears, and turnover cost, the kind of judgment call a dashboard of raw numbers doesn't make for the host.
  • →A reconciliation-grade audit trail — every derived number traceable back to its source email/CSV row, which is what makes the arrears-ledger and per-room P&L features actually trustworthy rather than another dashboard to doubt.
  • →PadSplit-native metrics — the ten-item wedge spec above, exposed as first-class views, not bolted onto an STR-shaped data model.
  • →Explicit "not a PMS" positioning — say it outright, the way Breezeway runs a standing "PMS vs. Breezeway" comparison page rather than hiding the distinction; it's what makes "read-only" read as a feature instead of an apology.
  • →A disclosed, zero-setup-fee claim — even Clearing, a narrower tool than a full PMS, still charges $199 to set up. Ours doesn't, and it's worth saying loudly.
06

Revised build order

PHASE 1

Table-stakes core, buildable today

Unified inbox (email + co-host read), multi-calendar view (iCal + email), team roles/permissions, portfolio dashboard. Plus the PadSplit wedge's data-confirmed pieces: arrears ledger, payout reconciliation, room-level roll-up — all sourced from already-verified email/CSV channels.

PHASE 2

Enrichment + differentiation

Draft-only guest-reply assist (human-click-send). QuickBooks/accounting read integration. Native mobile app (a real OwnerRez weakness to beat). Smart-lock/screening/guidebook integrations (Seam, Autohost, Touch Stay/Duve — all freely buildable, zero platform-approval risk). AI natural-language layer over the portfolio data, matching where the market is already headed (7 of 8 competitors now ship a named assistant). Per-room P&L / turnover-cost / utility-allocation wedge features, once a cost-capture mechanism exists (§04).

PHASE 3

Owned channel

A Cockpit-owned direct-booking site — the one channel Cockpit could fully control without any platform's permission — with Stripe Connect scoped specifically to that channel, never as a general OTA-payments substitute. Trust accounting as a named, sold feature once the reconciliation-grade audit trail is proven in production.
