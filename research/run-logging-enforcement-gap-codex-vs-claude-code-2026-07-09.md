# Run-logging enforcement gap: why Codex still skips durable run logs — 2026-07-09

## Trigger
User showed a Codex CLI screenshot (worktree `<HOME>/.codex/worktrees/3759/taxahead-connector-platform`) where Codex admitted: "verified, but not properly run-logged yet" — it captured plan-check/verifier/Coder-retry/final-verdict results only in-thread, never wrote them to a durable `runs/...` file. Question: why does this keep happening, and what's the concrete fix.

## (1) Does RUN.md mandate durable run-logging? NO.
`~/Claude/loop/RUN.md` (read in full) is the **career/job-application loop contract** — TRIGGER/SCOPE/ACTION/BUDGET/STOP/REPORT for job-finding/applying. It never mentions `runs/`, a run log, or an error log anywhere. Confirmed by grep across the file: zero hits for "run log"/"runs/". This is consistent with the standing decision in `~/Claude/CLAUDE.md`: "RUN.md is career-loop-specific — only load it for job/career tasks." A general software build (like taxahead's connector platform) is governed by `loop-team/orchestrator.md`, not RUN.md.

## (2) Does orchestrator.md mandate it? YES, in prose — and it now has a MECHANICAL gate too.
- Step 7 (`orchestrator.md:405`): **"Done. Write a run log to `runs/<timestamp>/` containing: the brief, the spec, each iteration's diff + verdict + the Coder's decision log … and the final summary."**
- Step 1 (`orchestrator.md:90`): **"Persist each plan-check cycle to `runs/<timestamp>/plan_check_log.md`: `gap_type`, `broken_assumption`, `proposed_fix`, iteration number, outcome."**
- Step 7 also has 3 more mandatory close-out checkpoints: Research checkpoint (line 407), Browser UI checkpoint (line 423), Lessons checkpoint (line 436) — all "state explicitly ... or state 'no X this run'" pattern.

This WAS pure instructional prose with no enforcement — confirmed by the framework's own prior self-diagnosis, frozen in `loop-team/harness/testdata/frozen_fix_plan_evidence_gate_phase1.md:3561-3573` (2026-07-03 incident record): *"The 'write a run log' step is a convention, not a gate. `orchestrator.md` step 7 says 'write a run log to `runs/<timestamp>/`' at the end of a build, but nothing in `hooks/loop_stop_guard.py` (confirmed via direct grep — zero hits for `run_log`/`RUN_LOG`/`iteration_log`) actually checks that file exists before letting a turn end. **Confirmed via first-person catch**: this exact session's own `runs/2026-07-03_h-subagent-masking-1/` … contain only a `specs/` subfolder … no run_log.md, no iteration log, nothing — Oga (this session) skipped the closing step … in real time, while this exact gap was being diagnosed."*

**This gap was then closed** — `H-RUNLOG-LOGGING-GAPS-1` (`fix_plan.md:3677`, `fix_plan.md:4938` cross-refs), CLOSED 2026-07-03, commit `e81215d`. `hooks/loop_stop_guard.py` now has a `RUNLOG_MISSING` gate (read live at `hooks/loop_stop_guard.py` lines ~1093-1168): for every sub-agent dispatch this turn whose text matches the Verifier-detection regex AND whose paired `tool_result` contains `verdict:\s*pass`, it resolves the referenced spec-file path to a run directory and checks for a non-empty `run_log.md`/`RUN_LOG.md`/`iteration_log.md` (3 grandfathered names — a related open finding, item 3 in the same frozen incident record, notes there was no single canonical filename across ~12 `run_log.md` vs ~9 `RUN_LOG.md` folders). If missing, it appends a `RUNLOG_MISSING` violation that **blocks Claude Code's Stop hook** with the message: *"A post-build Verifier returned VERDICT: PASS this turn for a build referencing %s, but no non-empty run_log.md/RUN_LOG.md/iteration_log.md exists there. Per orchestrator.md step 7, write run_log.md … before considering this build done."* Independently re-verified 253 passed / 1 pre-existing unrelated fail at close.

**So for a genuine Claude Code session inside this framework, run-logging is now a real, mechanical, non-bypassable-by-forgetting gate** — not merely advisory text — same category as `H-LIVE-VERIFY-COVERAGE-1`/`LOOP-M8`/`LOOP-M9`.

## ADDENDUM 2026-07-09 — Checkpointing (mid-build crash-recovery persistence, distinct from run_log.md)

Coordinator follow-up: is INCREMENTAL mid-build state persistence (resume-after-crash) mandated, distinct from the post-hoc `run_log.md`?

### (1) Mandated? YES — but as a git-commit checkpoint, not a "resume marker" file.
`RUN.md`/`VERIFIER.md`/`fix_plan.md`: zero hits for "checkpoint"/"resume"/"resumable"/"crash recovery"/"durable state" as a build mandate (fix_plan.md's "resumable" hits are all about the unrelated `evals/resumable_runner.py` judge-call tool, not build checkpointing). `orchestrator.md`'s micro-step build loop (step 2) is the real mandate: *"green → **git checkpoint commit immediately** (checkpoint = a commit; every gate's 'last checkpoint' is HEAD)"* (`orchestrator.md:226`), plus rule 5: *"Never thrash past green: a previously-green state is committed before any further Coder dispatch touches the tree"* (`orchestrator.md:236`). The "Priority-ranked context handoff" section additionally states the run log itself is updated **incrementally, at each checkpoint** — *"At each micro-step checkpoint (item 2's per-step commit), Oga updates a running, priority-ordered summary in the run log"* (`orchestrator.md:245`) — so `run_log.md` is prescribed (in prose) to be a live document during the build, not written only at step 7. No `TaskList`/`TaskCreate` resumable-task-list mechanism is referenced anywhere in `orchestrator.md` or `fix_plan.md` (zero hits) — the taxahead project memory shows this used ad hoc, session-scoped, explicitly NOT guaranteed to persist ("re-create if the list didn't persist").

### (2) Distinct from RUNLOG_MISSING — a SEPARATE, real mechanical gate.
Re-read `hooks/micro_step_gates.py` with this question in mind (not `loop_stop_guard.py`'s `RUNLOG_MISSING`, which only checks final non-empty existence, never incremental updates — confirmed no timestamp/diff check in `_runlog_has_real_log`). `micro_step_gates.py` has its own two commit-forcing gates, both live-read:
- **Gate 1 "thrash-past-green"**: blocks Stop with *"The last verify was GREEN, a Coder was dispatched after it, and %s still has uncommitted code changes with no commit since the green run. Commit the green state (git checkpoint) before ending the turn."*
- **Gate 2 "step-size"**: blocks Stop with *"%d uncommitted changed code lines … (max %d per micro-step). Commit what you have (a WIP commit if red) before ending the turn — an uncheckpointed pile is how verified work gets destroyed."*

These are the actual crash-recovery mechanism: they force a real git commit — durable, resumable state — after every green micro-step (≤200 changed lines at a time), so a crash loses at most one micro-step's work, not the whole build. This is genuinely separate machinery from `RUNLOG_MISSING` (different module, different trigger condition — git diff/log state, not Verifier-dispatch-text scanning).

### (3) Enforcement: MECHANICAL, not just prose — with one real activation caveat.
Both gates are hook-enforced (`micro_step_gates.py`, called from `loop_stop_guard.py`'s Stop firing) — genuinely a "something that can say no," same tier as `RUNLOG_MISSING`. Caveat: activation requires Oga to have written `$LOOP_GATE_DIR/<session>_target` at run start (`orchestrator.md` step 0 — an unenforced prose step); until armed, `micro_step_gates.py` docstring says "everything else is a silent allow." So the mandate to *arm* the gate is still prose; the gate's *behavior once armed* is mechanical.

### (4) Real historical incident backing the need? YES, adjacent but real — not hypothetical.
`loop-team/learnings.md:770`, 2026-07-02: *"Two Claude Code processes on the same session_id (crash + `--resume`) silently corrupted a playbook edit"* — a crashed session auto-resumed while the original process never exited, both independently dispatching sub-agents into the same tree; an unreviewed second diff hunk landed in commit `c94918c`, caught only by chance before a later commit, reverted (`96693f8`). Confirmed **undetectable from inside today's hook architecture** for ordinary foreground `--resume` (research: `research/claude-code-duplicate-session-detection-2026-07-02.md`), corroborated by an open, unresolved Anthropic issue (#25295). This is not the exact "checkpoint didn't exist so progress was lost" scenario, but it IS a real crash-adjacent incident in this framework's own history, and it's the direct motivating case for the git-checkpoint discipline (recoverable last-known-good state via `HEAD`) — separately, `fix_plan.md:320`'s `run_in_background` reaping incident (Claude Code's background-task manager silently killing long scrape jobs mid-run, fixed by moving the job under `launchd`) is a second real "process died mid-work, needed durable state outside the session" incident, though for a scraper task, not a loop-team build per se.

### (5) Codex parity: same blindness as RUNLOG_MISSING, likely worse.
`micro_step_gates.py`'s Gate 1 ("thrash-past-green") scans transcript `tool_use` parts for `name` in `("task","agent","subagent")` — the identical Claude-Code-Agent-SDK-specific tool-name dependency already flagged for `RUNLOG_MISSING`. It is invoked from inside `loop_stop_guard.py`'s Stop handler, so it inherits the same unverified Codex payload-shape problem documented in `PORTABILITY.md`. Additionally, activation itself depends on `$LOOP_GATE_DIR/<session>_target` — a file written by Oga's own prose step-0 action, keyed by `session_id`; whether this file gets written/read consistently in a Codex session (different session-ID scheme, different working-tree/worktree conventions — confirmed Codex uses its own `~/.codex/worktrees/<hash>/<repo>` layout, distinct from Claude Code's direct-cwd model) is untested. **Net: checkpointing enforcement is very likely inert under Codex for the same structural reason as run-logging enforcement — no confirmed adapter, PORTABILITY.md's backlog item still open.** The underlying git-commit ACT itself is tool-agnostic (any agent that runs `git commit` produces the same recoverable state), so Codex COULD checkpoint correctly if instructed to — but nothing currently forces it to, the way `micro_step_gates.py` forces Claude Code to.
