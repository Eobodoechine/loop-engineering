# Domain brief: "pxpipe" — repo that feeds Claude pictures to save usage limits

Mode D domain research, dispatched by Oga, 2026-07-15. User's own description: "research

## Headline finding

**The name was not wrong.** `pxpipe` (GitHub: `teamchong/pxpipe`, npm: `pxpipe-proxy`)
is a real, actively-maintained, verified repository whose stated purpose is an almost
word-for-word match for the user's description: it renders bulky Claude Code context
(system prompt, tool docs, conversation history) as PNG **images** and feeds those
images to Claude instead of raw text, because image tokens are cheaper per character
than text tokens on the Anthropic API — i.e. it literally "feeds Claude pictures to
save on usage limits." This was confirmed by three independent primary sources (not
search snippets): a direct `curl` of the raw README off `raw.githubusercontent.com`,
the GitHub REST API (`api.github.com/repos/teamchong/pxpipe`), and the npm registry
(`registry.npmjs.org/pxpipe-proxy`) — all fetched directly, bypassing any AI-summarized
web-fetch layer, specifically because the summarized fetch surfaced an odd model
codename ("Fable 5") that warranted independent confirmation before trusting it.

## Q1: Does a real repo called "pxpipe" exist, and does it match the user's description?

**Answer:** Yes, confirmed live. `github.com/teamchong/pxpipe`. First line of its
actual README (fetched raw, not summarized):

> "Cut Claude Code's input tokens by rendering bulky context as images — the same
> system prompt, tool docs, and history, in a fraction of the tokens."

And the GitHub API's own repo `description` field (set by the maintainer, not
generated by search):

> "cut Fable 5 token usage by rendering text context as images"

Mechanism, quoted from the README:

> "An image's token cost is fixed by its pixel dimensions, not by how much text is
> inside it. Dense content (code, JSON, tool output) packs ~3.1 chars per image-token
> vs ~1 char per text-token on real Claude Code traffic. The reader is the same vision
> channel that Anthropic's computer use already relies on for screenshots. pxpipe is a
> local proxy that uses that channel for context: it rewrites the bulky parts of each
> request into compact PNGs before it leaves your machine."

This is the correct match for "feeds claude's pictures to save on usage limits" — it
converts context to images specifically as a token-cost hack, not a generic
screenshot/image compressor.

## Q4: How is it actually invoked/wired in — CLI, MCP server, hook, or proxy?

**Answer:** It is a **local reverse proxy that Claude Code is pointed at via the
`ANTHROPIC_BASE_URL` environment variable** — not an MCP server, not a Claude Code
hook, not a plugin registered through Claude Code's plugin system. Quoted from the
README:

> "The proxy intercepts `/v1/messages`, rewrites eligible bulk into image blocks,
> splices them back cache-friendly (static prefix preserved, prompt caching keeps
> working), and forwards."

It ships a local dashboard for visibility:

> "Dashboard at http://127.0.0.1:47821/: tokens saved, every text→image conversion
> side by side, kill switch, live model chips. Responses stream normally — pxpipe
> compresses the request only, never the model's output."

It can also be consumed as a **library** (not just a standalone proxy process) via the
`./transform`, `./measurement`, `./applicability` exports for anyone building a custom
integration, and it can be deployed to Cloudflare Workers instead of run locally.

**Source:** README raw, sections "Try it (30 seconds)" and "How it works" (quoted above); `package.json` `exports` map.

## Q6: Gotchas the user/Coder must know before installing

**Answer:**
1. **It is lossy for exact/verbatim data.** Quoted from the README's "The honest
   part" section: "It is lossy. Exact 12-char hex strings in dense imaged content:
   13/15 on Fable 5, 0/15 on Opus, and 0/15 on Sol — misses are silent
   confabulations, not errors. Byte-exact values (IDs, hashes, secrets) must stay
   text; recent turns do." This means secrets/API keys/hashes appearing deep in
   imaged (compressed) context risk being silently misread by the model, not just
   dropped — a correctness risk, not merely a cost tradeoff.
2. **Off by default for most models.** Quoted: "Model scope: default
   `PXPIPE_MODELS=claude-fable-5`. Sol, Opus 4.7/4.8, GPT 5.5, and Grok are opt-in
   only... not good enough as silent defaults for imaged context." So out of the box
   it only activates image-rendering for one specific model identifier; other models
   pass through as plain text unless explicitly enabled.
3. **Workload-dependent payoff, not a universal win.** Quoted: "Wins on token-dense
   content (~1 char/token), loses money on sparse prose (~3.5 chars/token); a
   profitability gate (calibrated on N=391 production rows) images only where the
   math wins." It has internal logic to skip imaging when it wouldn't help — but that
   also means the advertised 59-70% figure is a workload-dependent ceiling, not a
   guarantee.
4. **Native dependency (`@napi-rs/canvas`).** Not pure-JS; platform/arch prebuilt
   binary. Standard desktop/server platforms are typically covered, but this is the
   one dependency that could break on an unusual runtime (e.g. some restricted
   sandboxes or exotic architectures).
5. **Model codenames in the docs ("Fable 5", "Sol") are not names I can independently
   verify.** My training data cutoff is January 2026; this repo's docs (dated
   May-July 2026) use codenames like `claude-fable-5`, `gpt-5.6-sol`, `claude-opus-4-8`
   that post-date what I can confirm from prior knowledge. I flagged this explicitly
   as worth extra scrutiny before assuming they map onto real, currently-shipping
   Anthropic/OpenAI model IDs — but note the same terminology appears consistently and
   independently across three separately-fetched primary sources (raw README, npm
   package `keywords` array which includes `"fable-5"`, and raw GitHub commit
   messages via the API, e.g. "steering Agent subagent calls to fable instead of
   haiku") which is strong internal-consistency evidence this is the project's own
   real terminology, not a fetch artifact or injection — but it is still a claim from
   a single-author repo's own docs, not something I could cross-check against an
   Anthropic source.
6. **All benchmark numbers (SWE-bench Lite/Pro, GSM8K, arithmetic/gist-recall
   evals) are self-reported inside the repo's own `eval/` directory**, with its own
   caveat ("Small n; receipts in `eval/`") — I did not independently reproduce them.
   Treat the specific percentages as a starting claim to verify against your own
   session, not a fact.

**Source:** README raw, "The honest part" section (quoted above in full for items 1-3); `package.json` dependencies; commit message for v0.9.0 (quoted above, item 5).

## Constraints (for whoever installs this)

- Node.js >= 18, ESM (`"type": "module"`) — no CommonJS `require()`.
- Native dependency `@napi-rs/canvas` — verify prebuilt binary exists for the target
  OS/arch before relying on `npx pxpipe-proxy` working out of the box in an unusual
  environment (e.g. some restricted/sandboxed containers).
- Wiring requires setting `ANTHROPIC_BASE_URL=http://127.0.0.1:47821` before invoking
  the `claude` CLI — this is an environment-variable-level redirect of all Claude Code
  API traffic through the local proxy process, which must stay running for Claude
  Code to work at all while it's set.
- Default model allowlist is narrow (`claude-fable-5` only) — confirm what that model
  ID resolves to in the currently-installed Claude Code version before assuming it
  will activate for whatever model the user is actually running; if it's a mismatch,
  imaging silently won't trigger and there will be no savings (fails safe, but
  silently).
- Byte-exact secrets (API keys, hashes, IDs) should be kept out of anything that gets
  imaged, per the maintainer's own documented lossiness numbers (item 1 in Q6).
- Single-author project, 8 weeks old — reasonable adoption risk for anything beyond
  personal/local use; not yet a widely-audited or long-track-record tool.
