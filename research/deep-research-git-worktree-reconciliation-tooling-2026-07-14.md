## Question researched

Tools/repos (current as of 2026) for reconciling git worktrees faster in a workflow running many parallel Claude Code / AI-agent sessions, each in its own worktree, later merged back. Scoped to 7 concrete pain points already hit in this loop-team practice: (1) worktree/branch naming-collision avoidance, (2) safe/explicit base-ref selection, (3) content-aware merging of untracked files, (4) concurrent-session locking, (5) canonical-file drift/deployment-gap detection, (6) architectural VCS alternatives, (7) purpose-built AI-agent+worktree orchestrators.

## Synthesis

Of the 7 pain points, only sub-question 7 (purpose-built orchestrator tools) and sub-question 2 (base-ref correctness) came back with strong, adversarially-confirmed answers. Three tools are genuinely active right now for orchestrating parallel AI-agent sessions across git worktrees — **Claude Squad**, **CCManager**, **Conductor** — while two well-known names in this space are dead ends: **Crystal** was deprecated in Feb 2026 (successor: **Nimbalyst**), and **Vibe Kanban** is sunsetting (maintainer-announced, no commits since 2026-04-24 despite 27k stars). None of the three active tools solve collision-safe naming, session locking, or canonical-file drift — they only isolate work into a worktree per agent.

The single most load-bearing finding: **Claude Code's own native worktree tooling has [[feedback_worktree_baseref_gotcha]]'s exact bug as documented, default behavior** — confirmed against official docs and an independent live GitHub issue, not just inferred from one incident. It's fixable with an opt-in `settings.json` toggle. Plain, unwrapped `git worktree add` does **not** have this problem — it correctly defaults to local HEAD; the bug is introduced specifically by wrapper tooling (Claude Code's own `--worktree` flag/`EnterWorktree`, and separately, Conductor) that deliberately defaults to a remote-tracking ref instead.

Three of the seven pain points — collision-safe naming (1), concurrent-session locking (4), canonical-file drift detection (5) — returned **no verified tooling**. This is evidence of a gap in this search round, not proof nothing exists anywhere, but it does mean the existing manual practices (pre-dispatch `ps`/`lsof` checks, orchestrator.md's deployment-gate step) aren't redundant busywork — they're covering ground no off-the-shelf tool covers yet.

## Key findings by sub-question

### 1. Collision-safe worktree/branch naming — no tool found
No actively-maintained tool surfaced that generates collision-safe (timestamp/UUID/session-ID) worktree or branch names, or detects naming collisions across concurrently-running agent sessions at creation time. CCManager's configurable directory-naming patterns (`../{branch}` style, with sanitization) are the closest tangential match, but they only control *where* a worktree lives on disk, not name uniqueness across sessions. Note: Claude Code's own `--worktree` auto-naming (e.g. `bright-running-fox`-style) surfaced in source material but wasn't run through adversarial verification this round — worth a targeted follow-up if this becomes the priority pain point.

### 2. Base-ref correctness — confirmed bug, has a fix (High confidence, 3-0)
- **Claude Code's own worktree tooling (`--worktree`, `EnterWorktree`) defaults to branching from `origin/HEAD`**, refreshed via a fetch capped at 5 seconds when the last fetch is >24h stale, falling back to local HEAD only if no remote is configured or `origin/HEAD` can't be fetched/cached. This is documented default behavior, confirmed near-verbatim against official docs (code.claude.com/docs/en/worktrees), independently corroborated by a live user-filed issue (github.com/anthropics/claude-code/issues/41680) describing exactly this failure mode. This is Claude Code's own behavior, not just a risk introduced by wrapper tooling.
- **The fix is an opt-in settings.json toggle:** `{"worktree": {"baseRef": "head"}}`. Enum only accepts `"fresh"` (default, origin/HEAD) or `"head"` (local HEAD) — no arbitrary ref accepted. Confirmed against current official docs (medium confidence — 2-1 vote — because a related claimed regression/fix-version narrative around this setting did NOT survive verification, see Refuted below; the setting's basic existence and behavior is solid).
- **Conductor has the identical failure mode**: new workspaces branch from the repo's configured base branch via a remote-tracking ref *after* a fetch — "a new workspace always starts from the latest remote commit, even if your local checkout is behind" — with no documented setting to branch from local HEAD instead. Confirmed verbatim against Conductor's own docs (not marketing copy), corroborated by a real user-filed issue (anthropics/claude-code#48456).
- **Plain `git worktree add` does not have this problem.** With no explicit commit-ish/`-b`/`-B`/`--detach`, it creates the new branch from local HEAD — confirmed via git's own docs *and* live empirical reproduction on git 2.50.1, plus an adversarial edge-case test (worktree path name matching an unrelated divergent remote branch — no DWIM fallback found). **Practical implication: `git worktree add -b <branch>` directly, or Claude Code's `baseRef: "head"` setting, sidesteps the issue entirely** — the bug is opt-in wrapper behavior, not a git limitation.

### 3. Content-aware merging of untracked files — no tool found (confirms your script is still warranted)
`git merge-tree` — both classic (`--trivial-merge`) and modern (`--write-tree`) modes — never reads from or writes to the working tree or index; it's a pure tree-object operation with zero visibility into on-disk untracked files. It structurally **cannot** detect or replicate the "untracked working tree file X would be overwritten by merge" error (that error comes from a separate porcelain `git merge` code path). No actively-maintained third-party tool was found that already packages the "pre-stage untracked files whose hash matches the incoming blob" workaround. **`~/Claude/scripts/git-content-aware-merge.sh` ([[reference_git_content_aware_merge_script]]) isn't reinventing a solved wheel — this appears to be a genuine, currently-unfilled gap.**

### 4. Concurrent-session safety — no verified tooling; one candidate failed verification twice
No confirmed lockfile/reservation tool for claiming a git worktree for a session's duration. One candidate, **Sandcastle** (mattpocock/sandcastle — a TypeScript library for orchestrating AI coding agents in worktree-isolated sandboxes), was investigated for a claimed atomic-lock-file mechanism with PID liveness checks — **both the core locking-mechanism claim and the "structured JSON lock file" detail failed adversarial verification (0-3 both times).** Do not adopt or cite Sandcastle's locking as a working solution without independently re-confirming against its actual current source. Your existing manual `ps aux`/`lsof` precondition check remains the best-known approach.

### 5. Canonical-file sync / deployment-gap detection — no tool found
No candidate tool was even surfaced for this pain point (the "isolated worktree fix is verified but the canonical registered copy outside normal merge flow has diverged" problem from [[feedback_isolated_worktree_deployment_gap]]). This looks like a genuinely uncommon problem shape outside this loop-team's specific pattern (config files that reference file paths outside git's own merge graph). Orchestrator.md's step 6.6 deployment-gate remains the only known mitigation.

### 6. Broader architectural alternatives
- **Jujutsu (jj)** — confirmed (3-0) as the clearest structural alternative. `jj workspace add` is a native multi-working-copy mechanism backed by one repository, treated by multiple independent 2026 sources as the direct analog to `git worktree add`. Can colocate with an existing git repo for incremental adoption (no full migration needed). Real documented friction: workspaces can go "stale" and need `jj workspace update-stale`; a Claude Code GitHub issue notes friction specifically in colocated git+jj repos. Jj's fitness for pain points 1-5 specifically was not evaluated this round.
- **GitButler "virtual branches"** — surfaced prominently in sourcing but **not independently verified this round** (didn't make the cut of 25 claims adversarially checked out of 105 extracted — flag as sourced-not-verified, not confirmed). Worth a dedicated follow-up: a real team (Trigger.dev) publicly wrote about replacing per-agent git worktrees entirely with GitButler's model — multiple branches applied simultaneously inside *one* shared working directory instead of one worktree per agent (trigger.dev/blog/parallel-agents-gitbutler; docs.gitbutler.com/ai-agents/parallel-agents). GitButler's own blog also claims a benchmark ("vcbench") showing it beating both plain git and jj on AI-agent-driven VC tasks — this is vendor-reported, not independently verified.
- git-branchless, git-town, Graphite, git-spice were sourced but did not surface high-relevance, verifiable claims this round.

## What's directly applicable to this loop-team practice

1. **Actionable now:** set `{"worktree": {"baseRef": "head"}}` in Claude Code's settings.json if worktrees should default to local HEAD rather than `origin/HEAD` — this is the documented, opt-in fix for exactly the failure mode in [[feedback_worktree_baseref_gotcha]]. Still worth keeping the manual `git rev-parse HEAD` vs `origin/<branch>` pre-check as a safety net, since this is opt-in, not automatic, and doesn't warn on divergence.
2. **Not a gap to fill:** collision-safe naming, concurrent-session locking, and canonical-file drift detection have no off-the-shelf answer — the existing manual disciplines (pre-dispatch `ps`/`lsof`, orchestrator.md step 6.6, escape-branch re-checks) are currently the best-known mitigations, not stand-ins for a tool that should be adopted instead.
3. **Not a gap to fill either:** `git-content-aware-merge.sh` has no actively-maintained replacement — worth continuing to maintain it rather than looking for a tool to retire it in favor of.
4. **Worth evaluating, not yet adopted:** Claude Squad, CCManager, or Conductor could replace some of the bespoke `EnterWorktree`/`ExitWorktree` session-management scaffolding for the *isolation* half of the problem (spinning up a worktree per parallel agent) — but none of them solve reconciliation/merge-back, which is the actual ask here. Conductor specifically inherits the base-ref bug, so CCManager or Claude Squad would need the same manual base-ref check either way.
5. **Worth a dedicated look:** Jujutsu (`jj workspace add`), colocated with an existing git repo, as an incremental experiment for one low-stakes worktree — it's the only structural alternative that reached high-confidence verification this round.

## Round 2 follow-up (2026-07-14, same day)

**Mechanism:** `deep-research` skill / Workflow (`wf_7df7b098-351`), 103 sub-agents, 911 tool uses, ~6.5M sub-agent tokens, ~61 min wall-clock.

**Question:** (A) independently verify the GitButler/Trigger.dev "ditched worktrees" claim and GitButler's vcbench benchmark; (B) broaden the search for tools solving gaps 1 (collision-safe naming), 4 (session locking), 5 (canonical-file drift) beyond the AI-coding-agent niche into general DevOps/platform-engineering tooling.

### Thread A — GitButler, re-examined: real, but not a clean worktree replacement
- GitButler's "virtual branches" is real, but is explicitly **not** runtime isolation per GitButler's own docs — agents share one filesystem, dependency install, and app state; the docs themselves recommend falling back to separate worktrees specifically when agents might touch the same file.
- GitButler's own maintainer (Byron), on GitHub, confirmed overlapping edits "can definitely lead to problems related to races and interference" — the actual fix (native worktree/bare-repo integration) is tracked in an open, unshipped issue (#10677); this round could not confirm its current implementation status.
- The Trigger.dev "we ditched worktrees" case study is a genuine account (real company, author is their founder/CTO) but is a single account cross-posted twice (company blog + dev.to under the same byline), not independent corroboration by two parties. The stronger claim that they "replaced" worktrees outright was **refuted (1-2)** — their own post keeps worktrees as the explicit fallback for same-file conflicts. Net: a hybrid workflow, not a clean swap.
- vcbench (GitButler's own benchmark claiming to beat git and Jujutsu) doesn't actually bear on this question: self-published by GitButler with a disclosed conflict of interest, only 300 runs, no statistical testing, and it tests single-session command-editing only — zero mentions of concurrency, worktrees, or parallel agents anywhere in it. The specific "~60% faster / ~80% fewer commands" headline did **not** survive re-verification as a standalone fact — treat as vendor marketing, not evidence either way for this use case.
- **Bottom line: GitButler is not a validated replacement for a worktree-per-agent pattern. It trades directory/branch collisions for in-file races, and its own maintainer says the real fix for that isn't shipped yet.**

### Thread B — broader search for the three confirmed gaps
- **Naming-collision (gap 1) — confirmed as a real, reproducible bug, not just a theoretical risk.** Git's own source has zero real collision protection: worktree admin-dir naming is a bare `while(mkdir) counter++` retry loop, no UUID/PID/timestamp, no documented atomicity. A live 5-trial race test this round (two processes creating a worktree at the identical path simultaneously, git 2.50.1) produced **corrupted phantom entries in 2/5 trials** — fully registered in git's internal bookkeeping with no actual directory ever created on disk. Independently corroborated by real filed issues, including three against Claude Code itself (#55724, #34645, #55708).
- **Locking (gap 4) — right generic building block identified, but not yet packaged for worktrees.** `flock(1)` (standard ~20-year-old Unix tool whose entire job is "wrap a lock around a command") is the correct conceptual primitive — but a claim that it can lock a directory path directly was refuted; the workable pattern is a sentinel lockfile alongside the worktree, not `flock`-ing the directory itself. Sandcastle hit this exact problem in production and proposed a fix (PID/branch/timestamp lockfile) — problem-identification is solid, but the claimed shipped-and-tested fix did **not** survive independent re-verification (closing comment wasn't from a listed maintainer). Treat Sandcastle's fix as unconfirmed, not adoptable as-is.
- **Canonical-file drift (gap 5) — closest pattern still isn't a real match.** chezmoi's default behavior (compare live file to expected, prompt before overwriting, never silently clobber) is the closest transferable pattern, but nothing purpose-built exists for the specific "hook script referenced outside any git checkout, drifting from its git-tracked source" scenario.
- **Explicitly came up empty:** Ansible/Puppet-style IaC drift detection, distributed lock managers (etcd/Zookeeper/Redis-style), and distributed job-scheduler naming-collision patterns — zero surviving evidence either way. Genuinely unexplored by this search, not a confirmed dead end.

### Updated bottom line after round 2
Nothing in round 2 overturns round 1: gaps 1, 4, and 5 remain unfilled by any existing, independently-verified tool. GitButler is a real but partial, maintainer-acknowledged-incomplete alternative — not a drop-in fix, and its own case study still falls back to worktrees for the exact scenario (same-file edits) that matters most here. The one new actionable idea: **`flock`-style sentinel-lockfile locking is the right conceptual pattern for gap 4** — nobody's shipped a reusable wrapper for git worktrees specifically, which makes this a candidate for a small home-grown script (same spirit as `git-content-aware-merge.sh`) rather than something to keep hunting for externally.
